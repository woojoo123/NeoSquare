# 2계정 QA 체크리스트

NeoSquare의 최종 QA는 `계정 2개`, `브라우저 2개(일반 창 + 시크릿 창 또는 서로 다른 브라우저)` 기준으로 진행하는 것이 가장 안정적입니다.

권장 계정:
- A: `mina@neosquare.local`
- B: `jisu@neosquare.local`
- 공통 비밀번호: `demo1234!`

## 1. 사전 준비

- backend 실행 확인
- frontend 실행 확인
- `http://localhost:8080/api/health` 응답 확인
- 두 브라우저에서 각각 로그인 가능한지 확인
- 카메라/마이크 권한 허용
- 가능하면 두 창을 좌우로 나란히 배치

## 2. 공통 합격 기준

- 새로고침 후에도 요청, 예약, 알림, 피드백이 서버 기준으로 유지된다
- 다른 계정에서 한 행동이 상대 계정 화면에 실제로 반영된다
- 로비에서 다음 행동이 바로 보이고, 막히는 지점 없이 흐름이 닫힌다
- 에러가 나더라도 화면이 완전히 깨지지 않고, 최소한의 안내 문구가 보인다

## 3. 로그인과 로비 진입

### 체크
- A 로그인
- B 로그인
- 두 계정 모두 `/lobby` 진입
- 현재 사용자 정보가 상단에 올바르게 표시되는지 확인
- 로비 맵, 우측 패널, 공개 채팅 드로어가 정상 렌더링되는지 확인

### 기대 결과
- 두 계정 모두 로그인 직후 로비에 진입한다
- 상단 사용자 닉네임과 이메일이 현재 계정과 맞는다
- 빠른 시작 카드가 현재 상태에 맞는 문구를 보여준다

### 실패 시 먼저 볼 것
- backend 기동 여부
- 브라우저 토큰 꼬임 여부
- `/api/auth/me` 응답

## 4. 로비 실시간 동기화

### 체크
- A가 로비에서 이동
- B 화면에서 A 위치가 반영되는지 확인
- B가 A 아바타를 클릭해서 선택 상태가 되는지 확인
- 공개 채팅에서 A가 메시지 전송
- B가 같은 메시지를 보는지 확인

### 기대 결과
- 다른 계정 캐릭터가 로비에서 보인다
- 아바타 클릭 시 우측 패널이 `사람 중심` 상태로 바뀐다
- 채팅 메시지가 양쪽에서 동일하게 보인다

### 실패 시 먼저 볼 것
- `/ws` 연결 상태
- 두 계정이 같은 공간에 있는지
- 브라우저 콘솔에 websocket 오류가 있는지

## 5. 멘토링 요청 흐름

### 체크
- A가 B를 선택하고 멘토링 요청 전송
- A 화면에서 `내 진행` 탭으로 자연스럽게 이동하는지 확인
- B 화면에서 `받은 요청` 탭 또는 알림에 요청이 보이는지 확인
- B가 요청 수락
- A와 B 화면에서 세션 입장 버튼이 열리는지 확인

### 기대 결과
- 요청 생성 후 A는 보낸 요청 상태를 바로 볼 수 있다
- 수락 후 B는 세션 입장 가능 상태를 본다
- A는 알림 또는 활동 패널에서 요청 수락 상태를 확인할 수 있다

### 실패 시 먼저 볼 것
- 요청 생성 API 응답
- 알림 목록 응답
- 활동 패널 탭 자동 전환 여부

## 6. 요청 기반 세션

### 체크
- 요청 세션 입장
- 채팅 입력/수신 확인
- 카메라/마이크 준비
- 영상 연결 시도
- `연결 다시 시도` 버튼 동작 확인
- 세션 종료

### 기대 결과
- 세션 페이지가 정상 진입한다
- 채팅과 signaling 상태가 보인다
- 영상 연결 실패 시에도 재시도 버튼과 안내 문구가 보인다
- 종료 후 로비로 복귀한다

### 실패 시 먼저 볼 것
- 카메라/마이크 권한
- `VITE_WEBRTC_*` 환경변수
- websocket signaling 연결 상태

## 7. 요청 기반 피드백과 알림

### 체크
- 로비 복귀 후 피드백 카드 노출 확인
- A 또는 B가 평점, 요약, 피드백 저장
- 기록 탭에서 저장 결과 확인
- 요청 수락 알림 읽음 처리

### 기대 결과
- 피드백 저장 후 기록 탭에 즉시 반영된다
- 다시 열어도 같은 피드백이 서버 기준으로 조회된다
- 읽음 처리 후 알림 목록 상태가 갱신된다

### 실패 시 먼저 볼 것
- 피드백 저장 API 응답
- `by-request` 조회 응답
- 알림 읽음 API 응답

## 8. 예약 흐름

### 체크
- A가 B에게 예약 생성
- A 화면에서 `내 진행` 탭 이동 확인
- B 화면에서 `받은 예약` 탭 확인
- B가 예약 수락
- 예약 시간이 입장 가능 범위에 들어오면 세션 입장 버튼 확인

### 기대 결과
- 예약 생성과 수락이 모두 서버 상태로 반영된다
- 입장 가능 시간에 세션 버튼이 열린다
- 요청 흐름과 달리 예약 상태 문구가 시간 기준으로 바뀐다

### 실패 시 먼저 볼 것
- 예약 시간이 과거인지
- 예약 상태가 `ACCEPTED`인지
- 예약 입장 가능 계산 문구가 보이는지

## 9. 예약 기반 세션과 피드백

### 체크
- 예약 세션 입장
- 채팅과 영상 연결 시도
- 세션 종료
- 로비 복귀 후 예약 피드백 저장
- 기록 탭에서 예약 기반 피드백 확인

### 기대 결과
- 예약 세션도 요청 세션과 같은 방식으로 채팅/영상 흐름이 열린다
- 종료 후 피드백 카드가 `예약 멘토링` 기준으로 보인다
- 기록 탭에서 요청/예약 피드백이 함께 보인다

## 10. 공간과 스터디 흐름

### 체크
- 두 계정이 같은 공간으로 입장
- 공간 안에서 아바타 클릭
- 채팅/멘토링 요청/예약 제안 버튼 확인
- 스터디 라운지에서 스터디 세션 생성
- 반대 계정으로 참가 후 스터디 세션 입장
- 스터디 세션 채팅과 영상 연결 시도

### 기대 결과
- 공간 페이지에서도 사람 선택과 상호작용이 이어진다
- 스터디 세션은 생성, 참가, 입장, 종료까지 닫힌다
- 스터디 세션 채팅과 영상 워크스페이스가 열린다

## 11. 재접속과 경계 상황

### 체크
- 로비에서 한 계정 새로고침
- websocket 재연결 배너 표시 확인
- 재연결 후 사용자 목록 복구 확인
- 세션 페이지에서 연결 실패 후 다시 시도 버튼 확인

### 기대 결과
- 재연결 중 안내가 보인다
- 복구 후 로비가 완전히 비정상 상태로 남지 않는다
- 세션 페이지가 실패 상태에 머물지 않고 다시 시도 가능하다

## 12. 배포 전 차단 이슈

아래 항목은 하나라도 실패하면 배포 전에 다시 봐야 합니다.

- 로그인 후 로비 진입이 불안정하다
- 2계정 로비 위치/채팅 동기화가 안 된다
- 요청 또는 예약 생성 후 상대 계정에서 안 보인다
- 세션 입장이 안 되거나 종료 후 복귀가 깨진다
- 피드백 저장 후 새로고침하면 사라진다
- 알림 읽음 처리나 전체 읽음 처리가 반영되지 않는다
- 예약 세션 또는 스터디 세션에서 채팅/영상 흐름이 막힌다

## 13. 점검 결과 기록 템플릿

간단히 아래 형식으로 남기면 됩니다.

```text
[통과] 로그인 / 로비 진입
[통과] 로비 실시간 위치 / 공개 채팅
[통과] 멘토링 요청 -> 수락 -> 세션 입장
[통과] 요청 세션 종료 -> 피드백 저장 -> 기록 조회
[통과] 예약 생성 -> 수락 -> 세션 입장 -> 예약 피드백
[통과] 알림 읽음 / 전체 읽음
[통과] 스터디 세션 생성 -> 참가 -> 채팅/영상
[확인 필요] TURN 실환경 검증
메모: Safari 미확인, 모바일 실기기 미확인
```
